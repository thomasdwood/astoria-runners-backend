# Phase 3.1: Refinements & Missing Features - Context

**Gathered:** 2026-02-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Address four accumulated TODOs: replace hardcoded route categories with admin-defined categories, add start location field with configurable default, show recurring event instances on admin events page, and support complex recurring patterns (nth-weekday, biweekly) using RRule. All changes build on existing Phase 3 infrastructure.

</domain>

<decisions>
## Implementation Decisions

### Admin-defined categories
- Categories have three properties: **name**, **color**, and **icon** (emoji or icon)
- Categories sorted **alphabetically** (no manual sort order)
- Category management lives on a new **admin settings page** (not a standalone page or modal from routes)
- Category deletion behavior: **Claude's discretion** on safest approach (block deletion vs reassign)
- Existing hardcoded categories (Brewery Run, Coffee Run, Brunch Run, Weekend) become seed data in the new categories table

### Start location & defaults
- Add `startLocation` field to routes and events
- Admin-configurable default start location (initial value: "Astoria Park Track"), managed on the **settings page** alongside categories
- When end location is left blank, **auto-copy start location** (most runs return to start)
- Start location input: **free text with autocomplete** suggesting previously-used locations
- Make `endLocation` non-required on routes and events

### Recurring events on admin page
- Recurring instances show **mixed in with one-off events**, with a visual badge/icon indicating they're from a recurring template
- Admin can **edit a single instance** (creating an exception) without affecting the template or other instances
- System generates recurring instances **8 weeks ahead**
- Two distinct actions for recurring instances:
  - **Cancel**: instance stays visible on public calendar as cancelled (strikethrough/greyed out)
  - **Delete**: instance removed entirely, doesn't show on calendar

### Complex recurring patterns
- Support **nth-weekday** patterns: 1st Monday, 2nd Wednesday, last Thursday, etc.
- Support **biweekly** patterns: every other Tuesday, etc.
- **Natural language preview** in the form UI: plain-English description ("Every 2nd Wednesday of the month") plus list of next 3 upcoming dates
- Structured dropdowns: frequency (weekly/biweekly/monthly) → day picker → for monthly: which occurrence (1st, 2nd, 3rd, 4th, last)
- **Auto-migrate** existing simple weekly templates to the new pattern format — one unified system
- Recurring templates support an **optional end date**, default to running indefinitely

### Event conflict handling
- When creating a one-off event that conflicts with a recurring instance (same time-of-day window — e.g., both morning or both evening), admin is prompted: **replace** (default) or **keep both**
- Replacing **deletes** the recurring instance (not cancel) — the one-off takes its place entirely

### Claude's Discretion
- Category deletion strategy (block vs reassign to fallback)
- Time-of-day window definition for conflict detection (e.g., morning = before noon, evening = after 5pm)
- Loading skeleton and empty state designs
- Exact badge/icon treatment for recurring instances on admin page
- Settings page layout and section organization

</decisions>

- Cancelled recurring instances should be visually distinct on public calendar (strikethrough or greyed out), not hidden
- Autocomplete for start location should draw from previously-used locations across all routes and events

</specifics>
