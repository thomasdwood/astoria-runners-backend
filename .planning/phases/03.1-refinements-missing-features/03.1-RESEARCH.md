## Summary

This phase adds four interconnected features to an existing Express + PostgreSQL + Drizzle ORM + React + TanStack Query application. The codebase is complete through Phase 4 with well-established patterns: Drizzle schema files in `src/db/schema/`, Zod validation in `src/validation/`, service layer in `src/services/`, Express routes in `src/routes/`, and React pages + hooks in `client/src/`. All new work follows these same patterns closely.

The most architecturally significant feature is **admin-defined categories**, which requires replacing a PostgreSQL enum (`route_category`) with a `categories` table. This is a breaking schema change: the enum must be removed, a FK column added, and both `routes` and `recurringTemplates` (which inherit categories from routes) must be migrated. Drizzle migrations handle this but ordering is critical. Existing categories become seed data.

The **complex recurring patterns** feature expands the existing `rrule.js` usage (already installed, v2.8.1 confirmed). The library already supports `bysetpos` for nth-weekday and `interval: 2` for biweekly. The existing `recurringTemplates` schema needs new columns (`frequency`, `interval`, `bySetPos`, optional `endDate`) and the `buildRRule` helper must be extended. The `count`-based approach currently in the schema is replaced by unbounded rules with 8-week lookahead via `.between()`. The `toText()` method does NOT handle `bysetpos` well â€” natural language must be hand-generated.

**Primary recommendation:** Implement in four independent vertical slices (categories, start location, events page recurring, complex patterns), each touching DB â†’ API â†’ UI. The settings page is a shared dependency between categories and start location â€” build it first within those slices.

### Pattern 2: Replacing pgEnum with FK

The existing `route_category` pgEnum in `src/db/schema/routes.ts` must be dropped and replaced with a `categoryId` FK column. This is a multi-step migration:

1. Add `categories` table with seed data
2. Add nullable `categoryId` FK column to `routes`
3. Backfill `categoryId` based on `category` string column
4. Drop old `category` column and enum
5. Make `categoryId` NOT NULL

Since this project uses `drizzle-kit push` (not migrate), steps can be done in a single schema change + manual SQL or in two pushes. Given the project uses `db:push` in Docker, the recommended approach is:

- Add `categories` table to schema
- Change `routes.ts`: remove `routeCategory` pgEnum, add `categoryId: integer('category_id').notNull().references(() => categories.id)`
- Run seed first (to populate categories), then push schema
- Update `recurringTemplates.ts` if it has a direct category reference (it does not â€” category is accessed through the route relation)

```typescript
// Modified routes.ts (no more pgEnum)
```

### Pattern 4: Extended RRule Builder (biweekly + nth-weekday)

The current `buildRRule()` in `recurringService.ts` only supports weekly. New version supports all three patterns. **Verified against rrule v2.8.1 installed in the project:**

```typescript
interface RecurrencePattern {
  frequency: 'weekly' | 'biweekly' | 'monthly';
  dayOfWeek: number;         // 0=Sun through 6=Sat
  monthlyPosition?: number;  // 1=first, 2=second, 3=third, 4=fourth, -1=last (only for monthly)
  startTime: string;         // HH:MM
  endDate?: Date;            // optional end date (until)
}

function buildRRule(pattern: RecurrencePattern): string {
  const dtstart = computeDtstart(pattern.dayOfWeek, pattern.startTime);
  const weekday = dayOfWeekToRRuleDay(pattern.dayOfWeek);

  let options: Partial<RRuleOptions> = { dtstart };

  if (pattern.frequency === 'weekly') {
    options = { ...options, freq: RRule.WEEKLY, byweekday: [weekday] };
  } else if (pattern.frequency === 'biweekly') {
    options = { ...options, freq: RRule.WEEKLY, interval: 2, byweekday: [weekday] };
  } else if (pattern.frequency === 'monthly') {
    options = { ...options, freq: RRule.MONTHLY, byweekday: [weekday], bysetpos: [pattern.monthlyPosition!] };
  }

  if (pattern.endDate) {
    options.until = pattern.endDate;
  }
  // No count â€” use .between() for 8-week lookahead instead of .all()

  return new RRule(options).toString();
}
```

**Verified output (tested against installed rrule 2.8.1):**
- Biweekly Tuesday: `RRULE:FREQ=WEEKLY;INTERVAL=2;BYDAY=TU`
- 2nd Wednesday monthly: `RRULE:FREQ=MONTHLY;BYDAY=WE;BYSETPOS=2`
- Last Thursday monthly: `RRULE:FREQ=MONTHLY;BYDAY=TH;BYSETPOS=-1`
- `.between(start, end, true)` generates instances correctly for all three patterns

### Pattern 6: Natural Language Recurrence Text

Since `rrule.toText()` does not handle `bysetpos` (outputs "every month on Wednesday" instead of "every 2nd Wednesday"), write a small helper:

```typescript
const ORDINALS = { 1: '1st', 2: '2nd', 3: '3rd', 4: '4th', [-1]: 'last' };
const DAY_NAMES = ['Sunday', 'Monday', 'Tuesday', 'Wednesday', 'Thursday', 'Friday', 'Saturday'];

export function formatRecurrenceText(pattern: RecurrencePattern): string {
  const dayName = DAY_NAMES[pattern.dayOfWeek];
  if (pattern.frequency === 'weekly') return `Every ${dayName}`;
  if (pattern.frequency === 'biweekly') return `Every other ${dayName}`;
  if (pattern.frequency === 'monthly') {
    const ordinal = ORDINALS[pattern.monthlyPosition as keyof typeof ORDINALS];
    return `Every ${ordinal} ${dayName} of the month`;
  }
  return '';
}
```

### Pattern 7: Cancelled Recurring Instance (New DB Concept)

The decisions require cancelled instances to remain visible on the public calendar (greyed out/strikethrough) while deleted instances disappear entirely. This requires materializing the instance as a DB event with a `isCancelled` flag:

- **Delete**: remove the virtual instance entirely (no DB row created) â€” done by storing a "cancelled override" or more simply by inserting a DB event with `isCancelled: true` and `status: 'cancelled'`
- **Cancel**: insert a DB event with `isCancelled: true` that overrides the virtual instance on the calendar

Schema addition to `events`:
```typescript
// Add to events table
isCancelled: boolean('is_cancelled').notNull().default(false),
```

The `mergeAndSortEvents()` helper in `calendarHelpers.ts` already handles deduplication between DB events and virtual recurring instances (DB events win by template+date key). Cancelled events simply need to be included in the merge with a flag that the frontend uses to render them differently.

### Pattern 8: Single Instance Edit (Exception Creation)

When admin edits a recurring instance (not the template), the system materializes that specific occurrence as a one-off DB event:

1. Create a new `events` row with `recurringTemplateId` set (links back to template)
2. Use the same date but different details (overriding the virtual instance)
3. The existing `mergeAndSortEvents()` will suppress the virtual instance on that date because a DB event with the same `recurringTemplateId` + date already exists

The edit form for recurring instances is the same `EventForm` used for one-off events, but pre-populated with the instance's inherited values.

### Pattern 9: Conflict Detection (One-off vs Recurring)

Time-of-day windows (Claude's discretion recommendation):
- **Morning**: before 12:00 PM
- **Afternoon**: 12:00 PM â€“ 5:00 PM
- **Evening**: 5:00 PM and after

On event create, check if a recurring instance exists for the same date within the same time-of-day window. If so, show a confirmation dialog before saving.

### Anti-Patterns to Avoid

- **Storing category as string after migration**: All references must use `categoryId` (integer FK). The old string `category` column is removed.
- **Using RRule count in stored RRULEs**: Drop `count` from all new RRULEs. Use `.between()` for generation. Existing templates with `count` will still parse correctly via `RRule.fromString()`.
- **Auto-copy endLocation server-side silently**: Do the auto-copy client-side in form `onSubmit` before sending. More transparent UX â€” user sees the field pre-filled.
- **Fetching all recurring instances for events page without date range**: Always use 8-week bounded range to avoid NĂ—infinity complexity.
- **Using rrule.toText() for bysetpos patterns**: It outputs incorrect text. Always use the custom `formatRecurrenceText()` helper.

## Common Pitfalls

### Pitfall 1: Drizzle Schema Migration Order

**What goes wrong:** Adding FK `categoryId` to `routes` before the `categories` table exists causes a constraint error.

**Why it happens:** `db:push` applies the full schema at once but table creation order depends on reference resolution. If Drizzle can't resolve the reference, it fails.

**How to avoid:** Add `categories` table to schema first (separate commit/push), then seed the category data, then add `categoryId` FK to `routes`. Alternatively: schema push with both tables simultaneously works if Drizzle resolves the dependency graph correctly â€” test this.

**Warning signs:** `ERROR: relation "categories" does not exist` during schema push.

### Pitfall 2: Breaking the pgEnum Drop

**What goes wrong:** Dropping the `route_category` enum while `routes.category` column still references it.

**Why it happens:** PostgreSQL does not allow dropping an enum that has dependent columns.

**How to avoid:** The column must be dropped (or altered to use the FK) before the enum is dropped. With Drizzle push, removing the enum from schema and replacing the column type simultaneously should generate the correct SQL â€” verify by running `db:generate` and inspecting the migration SQL before pushing.

**Warning signs:** `ERROR: cannot drop type route_category because other objects depend on it`.

### Pitfall 3: Category Filter Propagation

**What goes wrong:** Filtering by category throughout the app currently uses the string `category` field joined through routes. After migration to `categoryId`, all category-based queries must join `categories` table and filter on `categories.name` or `categories.id`.

**Why it happens:** The old system stores the category string on the route. The new system stores an integer FK.

**How to avoid:** Update `listRoutes()`, `listEvents()`, `listRecurringTemplates()`, `getAllInstancesInRange()`, and the calendar endpoint to accept `categoryId` or `categoryName` and apply the correct join.

**Warning signs:** Category filter returns all events after migration; or TypeScript errors on `routes.category` references.

### Pitfall 4: Existing RRULEs With Count

**What goes wrong:** Existing recurring templates have `count` in their RRULE strings (e.g., `RRULE:FREQ=WEEKLY;BYDAY=MO;COUNT=12`). After the 8-week lookahead migration, templates whose count has been exhausted return 0 instances.

**Why it happens:** `RRule.fromString()` respects `COUNT` â€” once 12 occurrences have elapsed, `.between()` returns empty.

**How to avoid:** Auto-migrate existing templates during the schema migration: strip `COUNT` from all existing RRULE strings and re-save them as indefinite rules. A one-time data migration script (can run as part of seed or as a separate migration step).

**Warning signs:** Existing recurring templates show no instances on the admin events page after the 8-week lookahead change.

### Pitfall 5: Autocomplete Location Suggestions Source

**What goes wrong:** The autocomplete queries `startLocation` values from routes and events, but `startLocation` is a new field that won't have data initially.

**Why it happens:** Feature needs existing data to suggest â€” cold start problem.

**How to avoid:** Seed the default "Astoria Park Track" into the suggestions. Also include `endLocation` values from existing routes/events as fallback suggestions initially. The API endpoint for suggestions can return deduplicated values from both start and end location columns.

**Warning signs:** Autocomplete shows no suggestions immediately after deployment.

### Pitfall 6: Zod Version Difference (Server vs Client)

**What goes wrong:** Server uses zod v4.x, client uses zod v3.x. Some APIs differ.

**Why it happens:** Both were installed separately and landed on different majors.

**How to avoid:** Do not share Zod schemas between server and client (they already don't). Keep server validation in `src/validation/` and client validation in form components. New category and settings Zod schemas follow this existing split.

**Warning signs:** `z.enum()` or `.array()` behavior differences in new shared code.

### Pitfall 7: toText() for bysetpos

**What goes wrong:** Using `rrule.toText()` for the natural language preview outputs "every month on Wednesday" for the 2nd Wednesday pattern.

**Why it happens:** rrule's `toText()` does not map `bysetpos` to ordinal language. Verified against v2.8.1.

**How to avoid:** Always use the custom `formatRecurrenceText()` helper for UI display. Never call `rule.toText()` in the recurring form preview.

### Category Badge (dynamic, post-migration)

After migration, `CategoryBadge` receives a `Category` object with `color` and `icon` instead of a hardcoded string. The component needs to map Tailwind color names to CSS classes dynamically:

## Open Questions

1. **Category deletion: block vs reassign**
   - What we know: Routes reference categories via FK; deleting a category with existing routes would violate FK constraint
   - What's unclear: User preference not stated (deferred to Claude's discretion)
   - Recommendation: Block deletion if any routes reference the category. Return 409 with count of referencing routes ("Cannot delete: 3 routes use this category"). This is safer than reassigning to a fallback (which would silently mutate data). The admin can manually reassign routes first.

2. **Tailwind dynamic colors for categories**
   - What we know: Admin defines `color` as a string; Tailwind purges dynamic class names
   - What's unclear: How many colors to support? Any non-Tailwind colors?
   - Recommendation: Offer a fixed set of 8-10 Tailwind color names (amber, orange, emerald, blue, purple, red, pink, teal) as a select dropdown in the category form. Pre-populate the colorMap in constants. This avoids the safelist problem entirely.

3. **Existing RRULE migration (count â†’ indefinite)**
   - What we know: Existing templates have `count` in their RRULEs; the 8-week approach requires removing count
   - What's unclear: Should this be a one-time script or part of the seed?
   - Recommendation: Add a migration step in the seed script (or a separate `db:migrate-rrules` script) that strips `COUNT` from all existing `recurringTemplates.rrule` values and re-saves them. Simple string manipulation: remove `;COUNT=\d+` from rrule string.

4. **Settings page: GET /api/settings access control**
   - What we know: Default start location needs to be readable by the event/recurring forms
   - What's unclear: Should GET /api/settings be public or auth-required?
   - Recommendation: Make GET /api/settings public (like GET /api/categories) so the event creation form can pre-fill the default start location without requiring authentication. PATCH /api/settings remains auth-required.
