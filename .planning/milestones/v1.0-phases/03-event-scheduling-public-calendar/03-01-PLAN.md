Create `src/services/eventService.ts` following the pattern from routeService.ts:

- `createEvent(data)`: Insert event with routeId, startDateTime (as Date object), endLocation, notes. Return event with joined route data (use db.query.events.findFirst with `with: { route: true }`). If route does not exist, return `{ error: 'route_not_found' }`.
- `getEventById(id)`: Return single event with route data, or null.
- `listEvents(filters?)`: Accept optional filters: category (string), start (Date), end (Date). Use Drizzle `and()` to combine: `gte(events.startDateTime, start)`, `lte(events.startDateTime, end)`. For category filter, JOIN routes table and filter `eq(routes.category, category)`. Order by startDateTime ascending.
- `updateEvent(id, data)`: Optimistic locking via version in WHERE clause (same pattern as routeService.updateRoute). Return discriminated union: `{ event }` or `{ error: 'not_found' | 'conflict' }`. Auto-increment version, set updatedAt to NOW().
- `deleteEvent(id)`: Return `{ success: true }` or `{ error: 'not_found' }`.

Note: For listEvents with category filter, use db.select() with explicit JOIN on routes table rather than relational query, since relational queries don't support filtering on joined table columns. Apply the category filter in the WHERE clause: `eq(routes.category, category as any)` -- type assertion is safe because Zod validates the enum at handler level (same pattern as routeService.listRoutes).

Create `src/services/recurringService.ts`:

- `createRecurringTemplate(data)`:
  1. Verify route exists (select from routes where id = data.routeId), return `{ error: 'route_not_found' }` if not
  2. Build RRULE string using rrule.js: `new RRule({ freq: RRule.WEEKLY, byweekday: [dayOfWeekToRRuleDay(data.dayOfWeek)], dtstart: computeDtstart(data.dayOfWeek, data.startTime), count: data.count || 12 }).toString()`
  3. Helper `dayOfWeekToRRuleDay(day)`: Maps 0-6 (JS Sunday=0) to RRule weekday constants (RRule.SU, RRule.MO, ..., RRule.SA)
  4. Helper `computeDtstart(dayOfWeek, startTime)`: Find the next occurrence of the given day of week from today, set hours/minutes from startTime string, return as UTC Date. Use date-fns `nextDay()` and `setHours()`/`setMinutes()` for this.
  5. Insert into recurring_templates table, return the created template with joined route data
- `getRecurringTemplateById(id)`: Return template with route data, or null.
- `listRecurringTemplates(filters?)`: List active templates, optionally filtered by route category via JOIN. Order by dayOfWeek, startTime.
- `updateRecurringTemplate(id, data)`: Optimistic locking update (same pattern). If dayOfWeek or startTime changes, rebuild the RRULE string. Return discriminated union.
- `deleteRecurringTemplate(id)`: Soft-deactivate by setting isActive=false (or hard delete if no materialized events reference it). Check if any events reference this template -- if yes, set isActive=false; if no events reference it, hard delete. Return `{ success: true }` or `{ error: 'not_found' }`.
- `getInstancesInRange(templateId, rangeStart, rangeEnd)`:
  1. Load template from DB
  2. Parse RRULE string with `RRule.fromString(template.rrule)`
  3. Use `rule.between(rangeStart, rangeEnd, true)` to get Date[] of occurrences
  4. Map each Date to a "virtual event" object: `{ recurringTemplateId, routeId, startDateTime, endLocation, notes, route }` (include route from template's joined data)
  5. Return array of virtual event objects
- `getAllInstancesInRange(rangeStart, rangeEnd, filters?)`:
  1. Load all active templates (optionally filtered by category via route JOIN)
  2. For each template, call `getInstancesInRange`
  3. Flatten results and sort by startDateTime ascending
  4. Return combined array

Import `RRule` from 'rrule'. Import `nextDay`, `setHours`, `setMinutes`, `startOfDay` from 'date-fns'. Use date-fns-tz `formatInTimeZone` only if needed for display formatting (likely not needed in service layer -- keep dates as UTC Date objects).
