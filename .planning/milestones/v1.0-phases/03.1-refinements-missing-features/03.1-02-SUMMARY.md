**Categories CRUD API with FK protection, settings key-value API with location suggestions, and route/event APIs migrated from category string enum to categoryId integer FK with startLocation support**

## Accomplishments
- Created categories CRUD API (`GET` public, `POST/PUT/DELETE` auth-protected) with in-use deletion protection returning 409 + routeCount
- Created settings API (`GET /api/settings`, `GET /api/settings/locations`, `PUT /api/settings/:key`) with deduped location suggestions from all route and event location fields
- Migrated route and event validation from `category` string enum to `categoryId` integer FK; added `startLocation` optional field; made `endLocation` optional everywhere
- Updated routeService to verify categoryId FK exists before create/update, use relational queries to return full category object with each route
- Updated eventService and recurringService to use nested relational queries (`route.with.category`) so category object flows through all event and calendar responses
- Fixed all downstream consumers (calendar route, recurringTemplates route, calendarHelpers) to use `categoryId` filter instead of `category` string
- Mounted `/api/categories` and `/api/settings` routers in `app.ts`

## Decisions Made
- Public GET endpoints for categories and settings (no requireAuth) so frontend forms and calendar can load data without user login
- Location suggestions via multiple Drizzle `selectDistinct` queries merged in JS (not raw UNION SQL) for type safety and Drizzle compatibility
- JS-side `categoryId` filtering after relational queries — simpler than SQL WHERE joins, acceptable for small category lists
- `CalendarEvent.category` changed from `string` to structured object to support rich UI display (color, icon, name)
